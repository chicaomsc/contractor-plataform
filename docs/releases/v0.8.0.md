# Sprint 7C — Landing Pública

## Objetivo

Implementar a landing pública definitiva do Contractor Platform para o tenant configurado por `NEXT_PUBLIC_COMPANY_SLUG`, mantendo o frontend genérico, multi-tenant e orientado pelos dados públicos da API Spring Boot.

## Direção visual aplicada

A direção "Obra em Ordem" foi aplicada com composição editorial, fotografia pública como protagonista quando disponível, fundos alternados, tipografia Barlow + DM Sans, pouco arredondamento, sombras mínimas e CTAs diretos.

## Estrutura da página

A rota `/` deixou de renderizar o preview técnico e agora apresenta:

1. Header
2. Hero
3. Barra de confiança
4. Serviços
5. Galeria antes/depois
6. Processo de trabalho
7. Sobre a empresa
8. Área geográfica atendida
9. Contacto
10. Footer
11. WhatsApp fixo no mobile quando houver número público válido

Seções opcionais são omitidas quando os dados públicos necessários não existem.

## Componentes

Componentes implementados em `frontend/src/features/public-site/components/`:

- `HeroSection`
- `TrustStrip`
- `ServiceSection`
- `GallerySection`
- `ProcessSection`
- `AboutSection`
- `ServiceAreaSection`
- `ContactSection`
- `MobileWhatsAppAction`
- `PublicLandingPage`
- `LandingStates`

O `SiteHeader` e o `SiteFooter` foram evoluídos para a versão definitiva da landing.

## Integração com API

O frontend continua a consumir exclusivamente:

- `GET /public/sites/{companySlug}`
- `GET /public/sites/{companySlug}/services`
- `GET /public/sites/{companySlug}/gallery`

O fluxo permanece:

```text
API DTO -> Mapper -> ViewModel -> Component
```

Os componentes da landing recebem ViewModels e não consomem DTOs brutos.

## Branding

O branding público continua a ser aplicado por tokens validados. Cores inválidas usam fallback seguro e não geram CSS arbitrário. A transformação segue:

```text
API branding -> validação -> tokens permitidos -> CSS custom properties
```

## Responsividade

A implementação é mobile-first, com:

- header com menu mobile acessível;
- hero reorganizado em mobile;
- galeria empilhada em ecrãs menores;
- comparador slider no desktop;
- CTA fixo de WhatsApp apenas em mobile e apenas quando há número válido;
- espaços e tipografia ajustados pelos breakpoints existentes.

## Acessibilidade

Foram aplicados landmarks semânticos, skip link existente, hierarquia de headings, nomes acessíveis para CTAs, menu mobile com `aria-expanded`, `aria-controls`, `hidden` quando fechado e encerramento por Escape, foco visível por tokens, listas semânticas, empty/error states identificáveis e comparador antes/depois com teclado.

## SEO

A metadata da rota pública passa a usar dados públicos quando disponíveis:

- title com nome comercial;
- description derivada do ViewModel;
- Open Graph base;
- canonical por `NEXT_PUBLIC_SITE_URL`;
- imagem social por logo pública quando disponível;
- locale `pt_PT`.

JSON-LD não foi adicionado porque os contratos públicos atuais não expõem dados suficientes para um `LocalBusiness` conservador sem campos fictícios.

## Performance

A landing usa `next/image`, dimensões/aspect-ratio estáveis, `priority` apenas na imagem principal do hero, lazy loading nas imagens restantes, cache TanStack Query existente, sem bibliotecas de animação, sem slider externo e com client boundaries limitados a TanStack Query, header mobile e comparador.

## Testes

A suíte frontend cobre:

- header desktop/mobile;
- footer;
- hero com e sem imagem;
- serviços ordenados e vazios;
- galeria com par completo e sem par;
- seções opcionais omitidas;
- contacto com e sem WhatsApp;
- erro parcial;
- ausência de conteúdo hardcoded do tenant;
- comparador antes/depois acessível;
- hooks, mappers, env e API client existentes.

## Validação visual

A validação visual foi feita localmente nos breakpoints solicitados após build/dev server. Itens verificados: hierarquia, espaçamento, contraste, ritmo, header, footer, CTA, fallback sem foto, empty states, before/after e ausência do preview técnico na rota pública.

## Conteúdo pendente

O resultado visual final depende da qualidade dos dados públicos cadastrados no backend:

- fotografias reais de obra;
- logo em alta resolução;
- texto institucional público;
- serviços ativos;
- pares antes/depois completos;
- contactos públicos válidos.

Sem esses dados, a landing degrada para fallbacks neutros e seções opcionais omitidas.

## Riscos

- Dados públicos incompletos reduzem a força comercial da landing.
- URLs remotas de imagem dependem de storage público estável.
- Ainda não há endpoint de formulário, então a seção de contacto usa apenas canais públicos existentes.
- Sem métricas públicas confiáveis, a barra de confiança não exibe números.

## Critérios de aceite

- Landing definitiva substitui o preview técnico.
- Dados vêm de ViewModels.
- Sem conteúdo específico da JR hardcoded nos componentes.
- Branding seguro aplicado por tokens.
- Header/footer definitivos.
- Comparador acessível reutilizado.
- SEO atualizado.
- Empty/error/loading states integrados.
- Lint, typecheck, testes e build frontend verdes.
- Testes backend verdes.
- Sem regra de negócio no Next.js.
