# Sprint 7D — Landing Polish & Production Readiness

## Objetivo

Elevar a landing pública implementada na Sprint 7C para um nível de produção, sem adicionar funcionalidades, endpoints, ViewModels ou alterações de backend.

## Escopo

- Revisão visual completa das secções existentes.
- Refinamento de tipografia, espaçamento, contraste e ritmo editorial.
- Microinterações subtis com CSS transitions.
- Validação responsiva em mobile, tablet e desktop.
- Revisão de acessibilidade, SEO e performance.
- Lighthouse local em build de produção.

- `SiteHeader`

- `GallerySection`

## Ajustes visuais

- Header com estados de hover, active e scroll mais discretos.
- Hero com composição mais equilibrada, imagem estabilizada e CTAs com presença mais forte.
- Serviços com tratamento editorial em linhas, reduzindo sensação de grid automático.
- Galeria com maior ritmo entre cabeçalho, comparador e itens secundários.
- Processo e contacto com labels adequados para fundo escuro.
- Footer com hierarquia e espaçamento mais claros.

## Tipografia e espaçamento

- Container mobile ajustado para maior respiro lateral.
- Hero com quebras de linha seguras em viewports estreitos.
- Leads com largura controlada e leitura mais confortável.
- Seções com ritmo vertical consistente.

## Acessibilidade

- Contraste AA reforçado para o token primário.
- Branding público passa por normalização de contraste antes de aplicar `--primary`.
- Token `--primary-on-dark` adicionado para labels e detalhes em fundos escuros.
- Estados de foco, hover e active preservam nomes acessíveis e navegação por teclado.

## SEO

- Fallback de `meta description` mantido no layout público.
- Ícone SVG local configurado para evitar 404 de favicon.
- Metadata dinâmica da rota pública preservada quando os dados da API estão disponíveis.

## Performance

- Sem novas bibliotecas.
- Sem Framer Motion.
- Sem sliders externos.
- Imagens continuam usando `next/image` e proporções estáveis.
- Bundle da rota pública manteve first load em torno de 152 kB no build final.

## Validação visual

Capturas finais geradas em:

- 320 px
- 375 px
- 768 px
- 1024 px
- 1440 px
- 1920 px

Problemas encontrados e corrigidos:

- Captura inicial de 320 px feita com Chrome headless desktop recortava o layout por viewport mínimo do browser. A validação foi repetida com Playwright e viewport real.
- Textos do hero em mobile foram protegidos com largura real de container e `overflow-wrap`.
- Primário de branding claro não atingia contraste AA com texto branco e fundo claro. A cor agora é escurecida de forma segura quando necessário.
- Labels em fundo escuro usavam primário pensado para fundo claro. Foi criado token específico para fundo escuro.
- Ausência de favicon local gerava erro de console no Lighthouse. Foi adicionado ícone SVG local.
- Lighthouse não detectava `meta description` na renderização analisada. Foi adicionado fallback estático no layout público.

## Riscos

- O Lighthouse foi executado com API local temporária e dados genéricos. Dados reais de produção podem alterar LCP se imagens remotas não tiverem dimensões e peso adequados.
- O branding escurece cores insuficientes para preservar contraste; a cor visual final pode divergir ligeiramente da cor cadastrada quando esta não cumpre WCAG AA.

## Critérios de aceite

- Landing refinada visualmente.
- Sem novas secções ou funcionalidades.
- Sem alterações no backend.
- Sem novos endpoints.
- Lighthouse dentro das metas.
- Gates de qualidade verdes.
- Documentação atualizada.
