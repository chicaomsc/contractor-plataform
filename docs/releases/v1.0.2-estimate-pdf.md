# Release v1.0.2 — Professional Estimate PDF

**Data:** 2026-07-18
**Sprint:** 10C — Professional Estimate PDF
**Escopo:** geração de PDF comercial de orçamento (`GET /estimates/{id}/pdf`) + integração mínima de download no dashboard. Sem link público, sem envio por email/WhatsApp, sem armazenamento permanente do ficheiro.

---

## Objetivo

Gerar um PDF de orçamento pronto para envio ao cliente final, usando exclusivamente dados já persistidos e calculados pelo backend (Sprint 10A) — nenhum valor financeiro é recalculado no renderer nem no frontend.

---

## Biblioteca Escolhida

**OpenPDF 1.4.2** (`com.github.librepdf:openpdf`, pacote `com.lowagie.text.*`). Licença dual **LGPL-2.1+ / MPL-2.0** — compatível com uso comercial fechado, sem custo, sem obrigação de abrir o código do produto. Decisão completa, alternativas rejeitadas (PDFBox, iText 5/7, Flying Saucer, OpenPDF 3.x) e justificativa em **[ADR-008](../adr/ADR-008-pdf-generation-strategy.md)**.

Motivo central: `PdfPTable` resolve nativamente os dois requisitos mais difíceis da sprint — repetição do cabeçalho da tabela em cada página (`setHeaderRows`) e proibição de cortar uma linha entre páginas (`setSplitRows(false)`) — sem exigir paginação manual.

## Estratégia de Snapshot

### Customer — snapshot mínimo implementado

`Estimate` já não depende do cadastro atual do `Customer` para os dados exibidos no documento. Novos campos frozen no momento da criação/reatribuição:

```
customerNameSnapshot, customerEmailSnapshot, customerPhoneSnapshot,
customerTaxNumberSnapshot, customerAddressSnapshot (Address embutido)
```

Regras aplicadas:
- Preenchidos em `EstimateService.applyCustomerSnapshot(...)`, chamado em `createEstimate` e sempre que `customerId` muda num `PUT /estimates/{id}`.
- `CustomerService.getAssignableCustomer(companyId, customerId)` (novo método) valida ownership/estado ativo **e** devolve os dados a snapshotar num único ponto — `assertAssignable` (Sprint 10A) passou a delegar para ele, mantendo compatibilidade com os chamadores existentes.
- O PDF usa exclusivamente o snapshot — nunca o `CustomerRepository` ao vivo. Verificado por teste de integração dedicado (`downloadPdf_containsCustomerSnapshot_unaffectedByLaterCustomerEdits`): editar o nome do cliente depois de gerado o PDF não altera o conteúdo de downloads seguintes.
- Snapshots nunca são aceites do frontend — construídos exclusivamente no backend a partir do `Customer` validado.
- Não é um sistema de versionamento: apenas os campos exibidos no documento são congelados, uma única vez por atribuição.

### Company/Branding — Alternativa A (dados atuais, sem snapshot)

Decisão deliberada, proporcional ao MVP: o PDF usa os dados **atuais** de `Company`/`Branding` (nome, logo, cor, contactos) no momento da geração — não um snapshot histórico.

**Justificativa:** ao contrário do Customer (um terceiro cujos dados históricos têm valor legal/probatório), Company é a identidade do próprio vendedor. Se a empresa corrige o telefone ou atualiza o logo, é desejável que o **próximo download** do mesmo orçamento reflita a informação atual — o mesmo comportamento de qualquer papel timbrado reimpresso. Nenhum PDF já baixado é alterado retroativamente (o ficheiro não é armazenado), apenas downloads futuros passam a refletir os novos dados.

Risco aceite e documentado: se no futuro for necessário fidelidade histórica total (ex.: reconstruir exatamente o PDF de uma data passada para fins de auditoria), será necessário adicionar um snapshot mínimo de Company/Branding — mudança de baixo custo dado o precedente já estabelecido para Customer.

## Migration e Backfill

**V9** `add_estimate_customer_snapshot.sql`:
- Adiciona as 5 colunas de snapshot (`customer_name_snapshot` NOT NULL, as restantes nullable, espelhando a nulabilidade do `Customer` de origem).
- Backfill via `UPDATE estimates ... FROM customers` — para orçamentos já existentes (Sprint 10A/10B), usa o cadastro atual do cliente como melhor aproximação disponível (não há histórico anterior a recuperar).
- `customer_name_snapshot` só é tornado `NOT NULL` **depois** do backfill, já que `customers.name` é garantidamente não nulo — sequência seguraacima de qualquer dado pré-existente.
- Nenhuma migration antiga foi editada.

## Estrutura de Classes

```
EstimatePdfController      (api)          → HTTP: Content-Type, Content-Disposition, delega tudo
EstimatePdfService         (application)  → ownership, carrega agregado + Company/Branding/Settings + logo, invoca factory/renderer
EstimatePdfDocumentFactory (pdf, puro)    → domínio → EstimatePdfDocument (formata, nunca calcula)
EstimatePdfRenderer        (pdf, puro)    → EstimatePdfDocument → byte[] (desenha, nunca calcula)
EstimateUnitLabels / EstimateStatusLabels (pdf) → mapeamento de apresentação centralizado
```

`EstimatePdfDocumentFactory` e `EstimatePdfRenderer` são classes simples (sem `@Component`), instanciadas diretamente pelo `EstimatePdfService` — mesmo padrão já usado para `EstimateCalculationService`/`EstimateStatusTransitionService` na Sprint 10A: sem Spring, sem banco, fáceis de testar isoladamente.

---

## Modelo do Documento

`EstimatePdfDocument` (record imutável) com sub-records `SellerInfo`, `EstimateMetadata`, `CustomerSnapshot`, `LineItem`, `FinancialSummary`. Todo campo já chega **formatado como String** (datas, moeda, percentuais, quantidades) — o Renderer não faz nenhuma conversão de dados, apenas desenha o que recebe.

---

## Responsabilidades

- **Application Service (`EstimatePdfService`):** valida ownership (`findDetailByIdAndCompanyId` — nunca busca por id sozinho), carrega Company/Branding/Settings, resolve bytes do logo via `StorageService.load` (novo método, tolerante a falha), monta o modelo e invoca o renderer. Transação `readOnly = true` — nunca grava.
- **Factory:** resolve textos e formatação (moeda a partir de `estimate.currency`, datas a partir de `Settings.dateFormat`/`locale`, cores validadas com fallback). Nunca recalcula `quantity × unitPrice`, `subtotal × vatRate` ou `total × upfrontPercentage` — usa exclusivamente os campos já persistidos.
- **Renderer:** desenho puro com OpenPDF — layout, paginação, tabelas, cabeçalho/rodapé repetidos. Sem acesso a banco, sem conhecimento de autenticação, sem regra financeira.

---

## Layout Implementado

Cabeçalho (logo opcional + dados da empresa) → título "Orçamento" + selo de rascunho/cancelado quando aplicável → metadados do orçamento → cliente (snapshot) → descrição → tabela de serviços → tabela de materiais → resumo financeiro → observações/condições (omitidos se vazios) → área de assinatura (nome/data/assinatura, sem assinatura digital) → rodapé com número, empresa e página, repetido em cada página.

Labels vazios nunca são renderizados (telefone, email, website, NIF, morada, descrição, notas, condições — cada um só aparece se preenchido).

## Estratégia de Paginação

`PdfPTable#setHeaderRows(1)` repete o cabeçalho da tabela em cada nova página. `setSplitRows(false)` + `setSplitLate(true)` garantem que uma linha nunca é cortada — a linha inteira migra para a página seguinte.

**Bug real encontrado e corrigido durante a inspeção visual desta sprint:** a configuração inicial tinha essa semântica invertida (`setSplitRows(true)`), o que cortava o texto de uma linha exatamente no limite da página. Corrigido e coberto por um teste de regressão dedicado (`render_neverSplitsARowAcrossPages`, verifica que marcadores no início/fim de cada linha longa aparecem sempre na mesma página).

Um segundo ajuste de espaçamento foi necessário: o espaçamento original entre secções empurrava a área de assinatura para uma segunda página quase vazia mesmo em orçamentos simples de 1 item + 1 material. Margens e espaçamentos foram reduzidos (ainda dentro de um padrão profissional) até o caso mínimo caber numa única página — confirmado visualmente.

## Branding Aplicado

Nome comercial (`Company.tradeName`, com fallback para `Company.name`), NIF, telefone, email, website, morada — todos condicionais. Logo: lido via `StorageService.load` (novo método na interface, implementado em `LocalStorageService`), nunca falha a geração — se ausente ou corrompido, o documento é gerado sem logo. Cor principal da `Branding` validada (regex hex + heurística de luminância percebida); cores ausentes, inválidas ou "demasiado claras para contraste" caem para um cinza-escuro neutro (`#2E2E2E`). A cor é usada apenas em elementos pequenos (régua do cabeçalho, cabeçalho de tabela, selo de rascunho/cancelado) — nunca em grandes áreas do documento.

## Formatação

- **Datas:** `Settings.dateFormat` + `Settings.locale` (já existentes desde a Sprint 10A) — sem introduzir configuração nova.
- **Moeda:** `NumberFormat.getCurrencyInstance(locale)` com `Currency.getInstance(estimate.currency)` — nunca assume EUR; formata `BigDecimal` diretamente (o `NumberFormat`/`DecimalFormat` do JDK tem um caminho de código dedicado para `BigDecimal`, sem conversão para `double`).
- **Percentuais/quantidades:** `BigDecimal.stripTrailingZeros().toPlainString()` — apresentação apenas, nenhuma alteração de valor.
- **Unidades:** `EstimateUnitLabels` centraliza os rótulos em português (mesmo enum `EstimateUnit` do domínio).

---

## DRAFT e CANCELLED

Ambos permitidos para download, sem alterar o agregado (`EstimatePdfService` é somente leitura). `DRAFT` mostra o selo "RASCUNHO — DOCUMENTO NÃO ENVIADO" na cor de destaque da empresa; `CANCELLED` mostra "CANCELADO" em vermelho. Nenhuma nova máquina de estados foi criada — a marcação é puramente visual sobre o `EstimateStatus` já existente.

## Endpoint

```
GET /estimates/{id}/pdf
```

- Autenticado (`hasRole('OWNER')`, mesmo mecanismo dos demais endpoints de `estimate`).
- `companyId` vem exclusivamente do `JwtPrincipal` — nunca da requisição.
- Consulta sempre com `(id, companyId)` — `EstimateRepository.findDetailByIdAndCompanyId` — nunca por id isolado.
- `Content-Type: application/pdf`.
- `Content-Disposition: attachment; filename="orcamento-{number}.pdf"` construído via `ContentDisposition.attachment().filename(name, UTF_8)` do Spring — emite automaticamente a variante `filename*=UTF-8''...` (RFC 5987/6266) quando necessário.
- Nome sanitizado: apenas `[A-Za-z0-9-_]` do número do orçamento.

---

## Segurança Multi-Tenant

Todas as consultas passam por `(id, companyId)`. Um `estimateId` de outra empresa nunca é resolvido — comportamento idêntico ao 404 já usado nos demais endpoints de `estimate` (nunca 403, para não confirmar a existência do recurso a outro tenant). Coberto por teste de integração dedicado (`downloadPdf_crossTenantEstimate_returns404`) e teste unitário (`generatePdf_estimateBelongsToAnotherCompany_throwsResourceNotFoundException`, que também verifica que nenhuma outra dependência é consultada nesse caso).

---

## ProblemDetail

Reaproveita o padrão existente — nenhuma classe de exceção nova: `ResourceNotFoundException` → 404 (estimate inexistente ou de outro tenant), `AuthenticationException`/ausência de token → 401, falha inesperada de renderização → 500 genérico já sanitizado pelo `GlobalExceptionHandler` (sem stack trace, sem detalhes da biblioteca). Falha ao ler o logo nunca propaga um erro — a geração prossegue sem logo.

## Integração no Frontend

Botão **"Baixar PDF"** em `/dashboard/estimates/[id]` (`DownloadPdfButton.tsx`), ação mínima:

- `adminApiRequestBlob` (novo, em `lib/api/admin-http-client.ts`) — mesmo padrão de autenticação/timeout/erro do `adminApiRequest` existente, mas devolve `Blob` + filename extraído do `Content-Disposition` (`filename*=UTF-8''...` com fallback para `filename="..."`).
- `useDownloadEstimatePdf` (TanStack Query `useMutation`) — sem invalidação de cache (download é somente leitura).
- Download disparado via `URL.createObjectURL` + `<a download>` sintético + `URL.revokeObjectURL` — nunca abre o PDF numa nova aba, nunca faz preview.
- Estados: `A gerar PDF` (loading, botão desabilitado, ícone animado), erro inline amigável, clique duplo prevenido enquanto `isPending`.
- Nenhuma biblioteca de PDF adicionada ao frontend. Nenhum cálculo financeiro no frontend.

## Riscos e Decisões Adiadas

- **Sem snapshot de Company/Branding** — decisão consciente (Alternativa A); reavaliar se fidelidade histórica total for exigida no futuro.
- **Sem cache do PDF gerado** — cada download reprocessa; aceitável para o volume do MVP (documento pequeno, geração em memória, sem I/O de rede além do logo já em disco local).
- **Acoplamento a `com.lowagie.text.*`** — migração futura para OpenPDF 3.x exigirá reescrever o Renderer (isolado do resto do domínio).
- **Nenhuma internacionalização** — textos fixos em português, mesmo padrão das sprints anteriores.
- **Nenhum template alternativo** — layout único; personalização de template fica fora do escopo do MVP.
