# Release v1.0.0 — Estimate Domain & API

**Data:** 2026-07-16
**Sprint:** 10A — Estimate Domain & API
**Escopo:** domínio e API REST de orçamentos (`Customer`, `Estimate`, `EstimateItem`, `Material`) — sem frontend, sem PDF.

---

## Objetivo

Iniciar o módulo de negócio principal da plataforma: orçamentos. O backend passa a ser a única fonte de verdade para todos os cálculos financeiros — o frontend futuro (Sprint 11) apenas envia dados e renderiza os resultados calculados pelo backend.

---

## Modelagem Final

### Customer

Referenciado apenas por FK (`Estimate.customerId`) — sem snapshot dos seus dados. Ver [Riscos e Decisões Adiadas](#riscos-e-decisões-adiadas).

- `id`, `companyId`, `name` (obrigatório), `email` (opcional), `phone` (opcional — **pelo menos um de `email`/`phone` é obrigatório**), `taxNumber` (opcional, não único), `address` (reutiliza o value object `Address`), `notes` (opcional), `active` (soft-delete), `createdAt`, `updatedAt`.

### Estimate (agregado raiz)

- Campos de negócio: `number`, `title`, `description`, `status`, `issueDate` (definido pelo backend, imutável), `validUntil`, `expectedStartDate`, `estimatedDurationDays`, `notes`, `terms`.
- Snapshots (nunca relidos após a criação): `currency`, `vatRate`, `upfrontPercentage`.
- Totais calculados pelo backend: `laborSubtotal`, `materialSubtotal`, `subtotal`, `vatAmount`, `total`, `upfrontAmount`, `remainingAmount`.
- `items: List<EstimateItem>` e `materials: List<Material>` — cascade `ALL` + `orphanRemoval`, sem repositórios próprios.

### EstimateItem

`estimate`, `serviceId` (opcional, referência não-normativa ao catálogo, sem FK de banco), `description`, `quantity`, `unit`, `unitPrice`, `total`, `displayOrder`.

### Material

Pertence **diretamente ao `Estimate`** (não ao `EstimateItem`) — sem catálogo global nesta sprint. Mesmos campos de linha que `EstimateItem`, mais `name`.

## Estratégia de Numeração

`ORC-{ano}-{sequência de 4 dígitos}`, ex: `ORC-2026-0001`. Contador atômico por `(company_id, year)` via `INSERT ... ON CONFLICT DO UPDATE ... RETURNING` — sem condição de corrida, sem lock explícito. Prefixo vem de `Branding.quotationPrefix` (fallback `ORC`). Detalhes completos e alternativas rejeitadas em **[ADR-007](../adr/ADR-007-estimate-numbering-strategy.md)**.

---

## Regras Monetárias e Arredondamento

- `BigDecimal` em todo o pipeline financeiro — nunca `double`/`float`.
- Valores monetários: scale 2. Percentuais (`vatRate`, `upfrontPercentage`): scale 2, `NUMERIC(5,2)`.
- Arredondamento: `RoundingMode.HALF_UP`, aplicado apenas no resultado final de cada etapa (linha, VAT, upfront) — não em intermediários, para evitar erro de arredondamento composto.
- `EstimateCalculationService` (`io.chicaodw.platform.estimate.domain`) é um serviço de domínio puro — sem `@Component`, sem Spring, sem acesso a banco — instanciado diretamente pela `EstimateService`. Testado exaustivamente em `EstimateCalculationServiceTest` (10 casos: sem materiais, sem mão de obra, ambos, VAT 0%/23%, upfront 0%/50%/100%, arredondamento).

```
laborSubtotal    = Σ EstimateItem.total
materialSubtotal = Σ Material.total
subtotal         = laborSubtotal + materialSubtotal
vatAmount        = subtotal × vatRate / 100
total             = subtotal + vatAmount
upfrontAmount     = total × upfrontPercentage / 100
remainingAmount   = total - upfrontAmount
```

## Snapshots

`currency`, `vatRate` e `upfrontPercentage` são copiados de `Settings` (e `Branding.quotationPrefix` para o prefixo do número) no momento da criação do `Estimate` e nunca mais relidos. Alterar `Settings` depois de um orçamento criado não afeta esse orçamento — coberto por `EstimateIntegrationTest.changingSettingsAfterCreation_doesNotAffectExistingEstimate`.

`Customer` **não** tem snapshot — é referenciado apenas por FK. Ver [Riscos e Decisões Adiadas](#riscos-e-decisões-adiadas).

---

## Transições de Status

`EstimateStatusTransitionService` (domínio puro) valida uma whitelist explícita:

```
DRAFT    → SENT, CANCELLED
SENT     → APPROVED, REJECTED, EXPIRED, CANCELLED
APPROVED → COMPLETED, CANCELLED
REJECTED, EXPIRED, CANCELLED, COMPLETED → (terminais)
```

Transição inválida (incluindo permanecer no mesmo status) → `ConflictException` → **409**.

---

## Regras de Edição

- `PUT /estimates/{id}` só é permitido com `status = DRAFT`. Fora disso → `BusinessRuleException` → **422**.
- `DELETE /estimates/{id}` só é permitido com `status = DRAFT` (exclusão física). Fora disso, usar `PATCH .../status` para `CANCELLED`. Regra explícita, sem ambiguidade.
- `Customer`: `DELETE /customers/{id}` é sempre soft-delete (`active = false`) — nunca exclusão física, pois `Estimate.customerId` referencia o cliente permanentemente.
- Um `Customer` inativo não pode ser associado a um `Estimate` (nem na criação, nem ao trocar o cliente de um orçamento existente).

## Endpoints

**Customers** (`hasRole('OWNER')`):
```
GET    /customers
POST   /customers
GET    /customers/{id}
PUT    /customers/{id}
DELETE /customers/{id}       -- soft delete (active = false)
```

**Estimates** (`hasRole('OWNER')`):
```
GET    /estimates?status=&customerId=
POST   /estimates
GET    /estimates/{id}
PUT    /estimates/{id}        -- somente DRAFT
DELETE /estimates/{id}        -- somente DRAFT
PATCH  /estimates/{id}/status
```

Sem paginação — segue o padrão já usado por `/services` e `/gallery` (listas simples, volume esperado baixo no cliente beta). Filtros `status`/`customerId` são opcionais e combináveis.

## Riscos e Decisões Adiadas

- **Customer sem snapshot:** `Estimate.customerId` é FK pura. Editar um cliente depois de criado um orçamento altera retroativamente o que é exibido ao consultar esse orçamento (nome, morada, etc.). Aceite conscientemente para o MVP — reavaliar quando o PDF (Sprint 11) tornar essa divergência visível ao cliente final. Mitigação futura de baixo custo: snapshot mínimo (nome + contacto) sem remodelar o agregado.
- **`EstimateItem.serviceId` sem FK de banco:** validado apenas na aplicação (`ServiceCatalogService.existsForCompany`). Escolha deliberada para nunca acoplar o histórico de um orçamento à existência futura de um `Service` no catálogo.
- **Sem paginação em `/estimates` e `/customers`:** aceitável para o volume do cliente beta; documentado como ponto de atenção caso o volume cresça.
- **`APPROVED → CANCELLED` permitido:** decisão explícita (a spec da sprint deixava isso em aberto) — um orçamento aprovado ainda pode ser cancelado antes da conclusão do trabalho.
