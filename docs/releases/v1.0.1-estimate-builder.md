## Componentes Reutilizáveis

- **StatusBadge** — traduz `EstimateStatus` para rótulos em português com cor por status (rascunho, enviado, aprovado, rejeitado, expirado, cancelado, concluído).
- **TotalsCard** — tile genérico label+valor, usado pelo resumo financeiro.
- **EstimateSummary** — monta o grid de `TotalsCard` a partir de `EstimateResponse`; nunca soma nada, apenas formata.
- **CustomerSelector** — pesquisa e seleciona um cliente ativo existente (`GET /customers`, filtrado no cliente).
- **QuickCustomerForm** — cria cliente inline (`POST /customers`) sem sair do wizard.
- **EstimateGeneralInfoForm** — título/descrição/validade/notas/condições (React Hook Form + Zod).
- **EstimateItemsEditor** / **MaterialsEditor** — adicionar, editar, excluir e reordenar (mover para cima/baixo — sem drag-and-drop, seguindo o padrão já estabelecido para Services/Gallery) linhas de item/material. Mostram apenas os campos de entrada (descrição, quantidade, unidade, preço unitário) — nunca um total calculado.
- **EstimateReviewStep** — revisão final antes de criar; lista itens/materiais sem qualquer total, com aviso explícito de que números/subtotais/IVA/total só existem após o backend criar o orçamento.
- **EstimateStatusControl** — oferece todas as transições possíveis e delega a validação inteiramente ao backend (409 em transição inválida).
- **EstimateNotEditableNotice** — bloqueia a UI de edição fora de `DRAFT` e explica o motivo.

## Fluxo do Wizard

`EstimateWizard` é um stepper local (sem biblioteca — nenhuma estava disponível no projeto) com 5 passos:

1. **Cliente** — `CustomerSelector` ou `QuickCustomerForm`.
2. **Informações gerais** — título, descrição, validade, notas, condições.
3. **Itens** — mão de obra/serviço.
4. **Materiais** — mesmo padrão dos itens.
5. **Revisão** — lista tudo o que foi preenchido, sem nenhum valor calculado, e cria o orçamento.

Itens/materiais entre etapas 3–5 vivem apenas em estado local do React (`useState`) até o `POST /estimates` final — nada é persistido incrementalmente.

---

## Fluxo de Criação

`EstimateWizard` → `useCreateEstimate()` → `POST /estimates` com `{ customerId, title, description, validUntil, notes, terms, items[], materials[] }`. O payload nunca inclui `number`, `total`, `subtotal`, `vatAmount` ou qualquer campo calculado — esses só existem na resposta. Ao suceder, o utilizador é redirecionado para `/dashboard/estimates/{id}`, onde os valores devolvidos pelo backend (número, subtotais, IVA, total, entrada, saldo) aparecem pela primeira vez.

Verificado manualmente end-to-end (backend + frontend reais, ver [Screenshots](#screenshots)): criação de cliente rápido, preenchimento dos 5 passos, criação do orçamento com número `ORC-2026-0001` gerado pelo backend e totais corretos (`45 × 12,50 € = 562,50 €` de mão de obra, `8 × 18,00 € = 144,00 €` de materiais, subtotal `706,50 €`, IVA 23% `162,50 €`, total `869,00 €`, entrada 50% `434,50 €`).

---

## Fluxo de Edição

`EstimateDetailPage` decide entre modo leitura e modo edição a partir de `estimate.status === "DRAFT"` — o único ramo condicional do frontend que espelha uma regra do backend, e apenas para UX (o `PUT` seria rejeitado com 422 de qualquer forma se o status mudasse entre o carregamento da página e o envio).

Em `DRAFT`: `EstimateGeneralInfoForm` + `EstimateItemsEditor` + `MaterialsEditor` pré-carregados com os dados atuais; "Guardar alterações" envia um único `PUT /estimates/{id}` com o formulário de informações gerais e os itens/materiais correntes (substituição completa, conforme o contrato do backend). Fora de `DRAFT`: `EstimateNotEditableNotice` explica o motivo e a UI de edição não é renderizada; apenas leitura (itens/materiais com o `total` devolvido pelo backend) e as ações de status/exclusão permitidas continuam disponíveis.

Exclusão (`DELETE /estimates/{id}`) só é oferecida em `DRAFT`, com confirmação em duas etapas.

---

## Fluxo de Mudança de Status

`EstimateStatusControl` lista todos os status exceto o atual como opções — o frontend **não reimplementa** a máquina de estados de `EstimateStatusTransitionService`. Uma transição inválida é enviada normalmente via `PATCH /estimates/{id}/status`; o backend responde 409 e o componente mostra o erro sem tentar adivinhar quais transições são "válidas" no cliente.

Verificado manualmente: `DRAFT → SENT` aplicado com sucesso, UI de edição e exclusão desapareceram imediatamente e `EstimateNotEditableNotice` foi exibida.

## Integração com Backend

Nenhum endpoint novo. Endpoints consumidos (todos já existentes desde a Sprint 10A):

```
GET    /customers
POST   /customers

GET    /estimates?status=&customerId=
GET    /estimates/{id}
POST   /estimates
PUT    /estimates/{id}
DELETE /estimates/{id}
PATCH  /estimates/{id}/status
```

DTOs espelhados 1:1 em `src/features/dashboard/types/{customers,estimates}.ts` via Zod — cada resposta é validada em runtime antes de chegar aos componentes, seguindo o mesmo padrão já usado por `admin.ts` (Company/Branding/Settings/Service/Gallery). Campos monetários do backend (`BigDecimal`) chegam como `number` no JSON (confirmado contra o contrato já usado por `Settings.defaultTaxRate`) e são validados com `z.number()`.

## Estados Implementados

`loading` (LoadingState reutilizado de `DashboardState.tsx`), `saving` (botões desabilitados + rótulo "A guardar"/"A criar"/"A excluir"/"A atualizar"), `success` (redirecionamento pós-criação; `SaveFeedback` para edição), `error` (`ErrorState` com retry para queries; banners inline para mutações), `retry` (botão "Tentar novamente" em toda query com erro), `empty` (`EmptyEstimates`, mensagem "nenhum orçamento corresponde aos filtros"), `conflict` (409 de transição de status surfaced explicitamente em `EstimateStatusControl`, sem crash e sem mensagem genérica).

## Confirmação: nenhum cálculo financeiro no frontend

Auditoria do código novo desta sprint:

- `EstimateSummary`, `TotalsCard` e a view de leitura de itens/materiais em `EstimateDetailPage` **apenas formatam** (`formatMoney`/`formatDate`, ambos em `utils/money.ts`) valores que já chegam prontos em `EstimateResponse`/`EstimateItemResponse`/`MaterialResponse`.
- `EstimateItemsEditor`/`MaterialsEditor`/`EstimateReviewStep` **nunca exibem um total de linha** durante a edição/revisão — apenas descrição, quantidade, unidade e preço unitário, exatamente os campos que o `CreateEstimateRequest`/`UpdateEstimateRequest` aceitam do cliente segundo o backend.
- `estimateItemFormSchema`/`materialFormSchema` (Zod) fazem apenas validação de formato (positivo, não-negativo) — nenhuma multiplicação, soma, percentual ou arredondamento.
- `EstimateStatusControl` não reimplementa `EstimateStatusTransitionService`: oferece todas as opções e deixa o backend decidir.
- Os payloads de `useCreateEstimate`/`useUpdateEstimate` (verificado em `estimate-hooks.test.tsx`) nunca incluem `total`, `subtotal`, `vatAmount`, `upfrontAmount`, `remainingAmount` ou `number` — apenas os campos de entrada que o backend espera.

Nenhuma regra de negócio foi duplicada.
